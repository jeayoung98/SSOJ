# Cloud Run 기반 채점 구조 검토

이 문서는 현재 Docker 기반 로컬 개발/검증용 worker를 기준으로, 상시 Docker 채점 서버 대신 Cloud Run 기반 구조를 왜 검토하게 되었는지 정리한 메모입니다. 운영 최종 구조 문서가 아니며, 비용과 초기 운영 부담 관점의 검토 배경을 중심으로 작성했습니다.

## 문서 범위

- 개인 프로젝트 관점의 비용 문제 정리
- 상시 VM 방식의 한계 정리
- Cloud Run 검토 이유 정리
- 현재 Docker executor를 그대로 Cloud Run에 올리기 어려운 이유 정리
- 운영 구조로는 추가 설계 필요

## 왜 이 구조를 검토했는가

개인 프로젝트에서는 "항상 켜져 있는 채점 서버"가 생각보다 비쌀 수 있습니다. 제출이 거의 없는 시간에도 VM을 계속 켜 두어야 하고, 피크를 버티기 위해 미리 어느 정도 스펙을 잡아야 하며, 그에 따라 비용과 운영 부담이 함께 올라갑니다.

작은 규모에서는 채점 요청이 몰리는 시간이 제한적일 수 있기 때문에, 상시 서버를 유지하는 방식이 실제 사용량보다 과한 선택이 될 가능성이 있습니다. 이 지점에서 Cloud Run 같은 서버리스 실행 모델을 검토할 이유가 생깁니다.

## 상시 VM 방식의 문제

### 1. 유휴 비용

- 제출이 없는 시간에도 VM 비용은 계속 발생합니다.
- Redis, DB, worker를 분리해 운영하면 채점 worker 인스턴스 비용도 고정비처럼 남습니다.
- 개인 프로젝트에서는 실제 트래픽보다 "켜 두기 위한 비용"이 더 크게 느껴질 수 있습니다.

### 2. 피크 대비 과한 스펙

- 채점은 순간적으로 CPU와 메모리를 많이 쓰는 작업이라 피크를 기준으로 VM 스펙을 잡기 쉽습니다.
- 하지만 대부분의 시간에는 그 스펙을 다 쓰지 못할 수 있습니다.
- 결국 낮은 평균 사용률에도 고정 비용을 지불하는 구조가 됩니다.

### 3. 운영 부담

- VM 유지, 패치, 로그 확인, 장애 대응, 오토스케일링 전략을 직접 챙겨야 합니다.
- Docker 실행 환경과 worker 프로세스를 안정적으로 관리해야 합니다.
- 개인 프로젝트에서는 기능 개발보다 운영 유지 비용이 더 부담이 될 수 있습니다.

## Cloud Run을 검토한 이유

### 1. scale-to-zero

- 요청이 없으면 인스턴스가 0까지 내려갈 수 있습니다.
- 제출이 없는 시간대 비용을 크게 줄일 수 있습니다.
- 개인 프로젝트에서 가장 매력적인 포인트는 "유휴 상태 비용 절감"입니다.

### 2. 요청 기반 과금

- 실제 요청이 들어와 컨테이너가 실행된 만큼 비용을 내는 구조입니다.
- 사용량이 적은 초기 단계에서는 고정비보다 변동비가 더 유리할 수 있습니다.
- "항상 켜 두는 서버"보다 제출량에 가까운 비용 구조를 기대할 수 있습니다.

### 3. 초기 운영 적합성

- VM을 직접 관리하는 부담이 줄어듭니다.
- 배포, 인스턴스 생성, 기본 스케일링을 플랫폼에 맡길 수 있습니다.
- 프로젝트 초기에 트래픽이 크지 않다면 단순한 운영 모델로 시작하기 좋습니다.

## 하지만 현재 Docker executor 구조를 그대로 Cloud Run에 올리기 어려운 이유

현재 worker는 단순히 "Spring 애플리케이션이 돌아간다" 수준이 아니라, 로컬 또는 VM 안에서 직접 Docker를 호출하는 구조입니다.

현재 executor 구조의 특징:

- `ProcessBuilder`로 `docker run` 실행
- 로컬 temp directory 생성 후 source file 작성
- `-v <temp>:/workspace` 형태의 volume mount 사용
- Redis polling 기반 상시 worker 모델
- 채점 중 timeout 대기와 프로세스 정리 로직 포함

이 구조는 Cloud Run에 그대로 옮기기 어렵습니다. 이유는 다음과 같습니다.

- Cloud Run은 기본적으로 요청 기반, stateless 실행 모델에 가깝습니다.
- 현재 구조처럼 "Cloud Run 내부에서 다시 Docker CLI를 실행하는 방식"은 전제가 다릅니다.
- 로컬 temp path를 만들고 Docker volume mount로 연결하는 방식도 그대로 사용하기 어렵습니다.
- Redis를 계속 polling 하는 상시 worker 모델과 Cloud Run의 요청 기반 실행 모델도 잘 맞지 않습니다.

즉, 이것은 단순히 VM을 Cloud Run으로 바꾸는 수준의 문제가 아닙니다.

## 핵심 판단

Cloud Run 검토는 충분히 의미가 있습니다. 특히 개인 프로젝트 관점에서는 유휴 비용을 줄이고 초기 운영 부담을 낮출 수 있다는 장점이 분명합니다.

하지만 현재 Docker executor 구조를 유지한 채 인프라만 Cloud Run으로 바꾸는 것은 어렵습니다. 이 경우 필요한 것은 단순 인프라 교체가 아니라 채점 실행 모델 자체를 다시 설계하는 아키텍처 전환입니다.

## 왜 아키텍처 전환이라고 보는가

현재 구조는 다음 전제 위에 서 있습니다.

- 상시 떠 있는 worker가 Redis queue를 polling
- worker 인스턴스 안에서 직접 Docker 실행
- 로컬 파일시스템과 프로세스 생명주기를 worker가 직접 관리

Cloud Run 기반 구조를 고려하면 보통 다음 방향으로 사고가 바뀝니다.

- 요청 또는 job 단위로 채점 실행
- stateless 실행 단위로 결과를 계산
- 실행 환경 제약에 맞는 별도 sandbox 또는 원격 실행 모델 검토
- queue 소비 방식과 채점 시작 방식 재설계

즉, worker를 "항상 떠 있는 Docker judge 서버"로 둘지, 아니면 "요청이 올 때만 실행되는 채점 단위"로 바꿀지부터 다시 정해야 합니다.

## 현재 시점의 결론

- 개인 프로젝트 비용 관점에서는 Cloud Run 검토가 충분히 타당합니다.
- 상시 VM은 유휴 비용과 운영 부담이 큽니다.
- Cloud Run은 scale-to-zero와 요청 기반 과금 덕분에 초기 운영에 유리할 수 있습니다.
- 하지만 현재 `docker run` 기반 executor를 그대로 옮기기는 어렵습니다.
- 따라서 Cloud Run 전환은 "배포 대상만 변경"하는 작업이 아니라 "채점 아키텍처를 다시 설계"하는 작업으로 봐야 합니다.

## 구조 구분표

| 구분 | 내용 |
| --- | --- |
| 현재 구조 | Spring worker가 Redis `judge:queue`를 polling 하고, 각 executor가 로컬/VM 안에서 직접 `docker run`을 호출해 채점 |
| 검토 구조 | Cloud Run 기반으로 요청 또는 job 단위 채점을 수행하고, 사용량 기반으로 scale-to-zero를 활용하는 구조 |
| 아직 미구현 | Cloud Run에 맞는 채점 실행 모델, Docker 대체 또는 원격 sandbox 방식, queue 소비 방식 재설계, 결과 저장까지 포함한 전체 아키텍처 전환 |
